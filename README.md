# DocForge

**Your thinking, structured. Your document, written.**

Most AI writing tools give you a blank box and a generate button. DocForge is different. It's a five-stage collaborative pipeline — ideation partner, claim validator, drafter, adversarial reviewer, editor — that draws out your thinking, challenges your assumptions, grounds every claim in evidence, and produces documents that actually hold up to scrutiny.

You're the author. DocForge is the collaborator that makes sure you don't skip the hard parts.

## How It Works

DocForge runs a five-stage pipeline. Each stage is a specialized agent with a distinct role and a cold context — they only know what they need to know. The ideation partner doesn't draft. The reviewer has never seen the brief. The editor doesn't second-guess the review findings. This separation is what produces documents that read like they were written by someone who thought hard, not generated by someone who prompted fast.

**The pipeline:**

```
Ideation Partner  →  Researcher  →  Drafter  →  Adversarial Reviewer  →  Editor  →  Polish
   (draws out          (validates      (writes        (cold read —           (revises    (final
   your thinking)       claims)        the doc)       no prior context)      to brief)    pass)
```

You just talk. DocForge reads the workspace, figures out which stage you're at, and routes automatically.

```
You:    "I want to write a strategy doc about treating AI as specialized
         engineers instead of one general assistant"

Claude: [Assesses complexity → asks clarifying questions one at a time]
Claude: [After 8-10 questions → produces workspace/brief.md]
Claude: "Brief saved. Add grounding files and do your own research, then
         we'll validate your claims."

You:    [Copy existing docs, system inventories, research to workspace/grounding/]
You:    "Ready, validate my claims"

Claude: [Reads brief + grounding files → validates every claim]
Claude: [Produces workspace/research.md with evidence + gaps]
Claude: "Your grounding is solid. Missing: KTLO volume data. Ready to draft?"

You:    "Draft the problem statement and proposed approach first"

Claude: [Sectional drafting → produces first sections of workspace/draft-v1.md]
Claude: "First sections ready. Review them, then we'll draft architecture."

You:    "Looks good. Draft the architecture section. Here's the agent
         definition I want included verbatim: [code block]"

Claude: [Drafts architecture with author-specified content → adds to draft]

You:    "Review the full document for substance"

Claude: [Cold read — no brief/research context → focused substance review]
Claude: [Produces workspace/review-v1.md with findings]

You:    "Accept 1,3,5. Reject 2. Also add a governance section — here's
         what I want in it: autonomy policy, security, kill switch..."

Claude: [Implements finding fixes + author-directed content → workspace/draft-v2.md]

You:    "Polish it"

Claude: [Final pass → workspace/FINAL.md]
```

No commands to remember. You just talk. DocForge figures out the stage and routes to the right agent.

## Why a Pipeline, Not a Prompt

A single AI session writing your strategy doc is like asking someone to be your brainstorming partner, fact-checker, author, critic, and editor simultaneously. The roles conflict. The critic who helped you brainstorm will unconsciously fill in gaps it already knows about. The author who did the research will defend its own claims.

DocForge separates these roles deliberately:

- **The Ideation Partner** asks questions before touching a keyboard. It doesn't write until it understands your argument, your audience, and what decision you want the reader to make.
- **The Researcher** tests your claims against evidence — and will tell you when confidence is LOW. It doesn't confirm your thesis; it stress-tests it.
- **The Drafter** writes from the brief and evidence package, not from the conversation. It knows the structure, follows the template, and grounds every claim in what was actually validated.
- **The Adversarial Reviewer** reads cold — no brief, no research, no prior context. It sees exactly what your real audience will see. It finds the gaps the author can't see.
- **The Editor** implements findings against the author's explicit direction. Accept this finding, reject that one, inject this content here.

The result is a document that has been genuinely challenged, not just generated.

## The Human's Role

DocForge is a **collaboration framework**, not a generation tool. The quality of the output is proportional to the quality of what you put in. A brain dump produces a thin brief. A thin brief produces a generic draft. Grounding files — your real data, your real system context, your real research — are what make the difference between a document that could have been written by anyone and one that only you could have written.

| What you provide | How it enters the pipeline |
|-----------------|---------------------------|
| Your raw thinking | Chat naturally — the ideation agent draws it out |
| Internal context (existing docs, inventories, metrics) | Copy files to `workspace/grounding/` |
| External research (industry data, competitive analysis) | Use ChatGPT Deep Research, Perplexity, Google Scholar, or whatever you prefer — save results to `workspace/grounding/` |
| Specific content (code, definitions, examples, tables) | Tell the editor to include it, or add to `workspace/grounding/author-content.md` |
| Judgment and direction | Accept/reject review findings, direct expansions, set priorities |
| Memory curation | Decide which learnings persist for future sessions |

For a short ADR, you might provide a brain dump and a few answers. For a Blueprint strategy document, expect to be actively involved throughout — detailed answers, grounding files, research, content injection, and review feedback.

## Slash Commands (Optional)

You don't need these — just chat. But if you prefer explicit control:

| Command | What it does |
|---------|-------------|
| `/df-ideation "brain dump"` | Start ideation |
| `/df-research` | Validate claims |
| `/df-draft` | Draft (or `sections:1-3` for sectional drafting) |
| `/df-review` | Adversarial review (or `structural` / `substance` / `audience`) |
| `/df-edit "feedback"` | Edit with feedback |
| `/df-polish` | Final polish |
| `/df-supplement "additions"` | Add context mid-pipeline |
| `/df-status` | Show pipeline state |

## Getting Started

1. Open this folder in your IDE with Claude Code
2. Tell DocForge what you want to write — brain dump welcome
3. Answer the ideation partner's questions (it earns this — the questions matter)
4. Add grounding files to `workspace/grounding/` when prompted
5. Follow the conversation through the pipeline
6. Your finished document is at `workspace/FINAL.md`

For a short ADR, the whole pipeline takes an hour. For a Blueprint strategy document, plan to be actively involved across multiple sessions — detailed answers, grounding files, research, content injection, and review feedback. The pipeline scales to the document's complexity.

## Project Structure

DocForge has no application code. It's entirely markdown — agent briefings, templates, rules, and commands — that turn Claude Code into a structured multi-agent writing system. The architecture is transparent and modifiable: every agent's behavior is defined in a readable briefing file.

```
docforge/
├── CLAUDE.md                    # Auto-routing orchestrator (always active)
├── DEVELOPMENT.md               # Instructions for modifying DocForge itself
├── .claude/
│   ├── commands/                # Slash command shortcuts (optional)
│   ├── agents/                  # Agent briefings
│   │   ├── ideation/AGENT.md
│   │   ├── researcher/AGENT.md
│   │   ├── drafter/AGENT.md
│   │   ├── reviewer/AGENT.md
│   │   ├── editor/AGENT.md
│   │   └── polish/AGENT.md
│   ├── templates/               # Document-type templates
│   ├── memory/MEMORY.md         # Cross-session learnings
│   └── rules/                   # Behavioral rules
└── workspace/                   # Active document session (gitignored)
    ├── brief.md
    ├── research.md
    ├── grounding/               # Your context files go here
    ├── draft-v*.md
    ├── review-v*.md
    └── FINAL.md
```
